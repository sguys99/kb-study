# lesson.md 표준 템플릿

각 토픽의 `lesson.md`는 아래 골격을 따른다. 대괄호 `[...]`는 채워 넣을 자리이며, 6개 필수 요소(학습 목표 / 완료 기준 / 이론+코드 실습 / 자주 하는 실수 / 출처 / 다음 토픽 링크)는 **하나도 빠뜨리지 않는다.**

순서·헤딩 구조는 토픽 성격에 맞게 조정해도 되지만, 6개 요소는 반드시 모두 존재해야 한다.

---

````markdown
# [토픽 번호와 제목 — 예: 1.6 Baseline Hybrid RAG (기준선)]

> **Phase [N] · 토픽 [NN]** · 이 토픽의 한 줄 요약 — [무엇을 만들고 왜 만드는가]

## 학습 목표

이 토픽을 끝내면 다음을 할 수 있다.

- [목표 1 — 행동 동사로. 예: Vector+BM25 하이브리드 검색기를 직접 구성한다]
- [목표 2]
- [목표 3]
<!-- 최소 3개. 각 목표는 측정 가능한 행동으로 쓴다(이해한다/안다 X → 만든다/비교한다/측정한다 O) -->

**완료 기준**: [한 줄. 예: `mix` 모드 답변에 인용 문서 3건이 붙고, Vector-only 대비 멀티홉 정답률이 올라가면 완료]

---

## 1. 왜 필요한가 — [동기]

[기존 방식이 어디서 무너지는지, 또는 앞 Phase 산출물의 어떤 한계를 이 토픽이 메우는지. 누적 스토리라인과 연결.]

## 2. [핵심 개념]

[직관 우선. 정의는 짧게. 그림이 필요하면 ./img/ 참조.]

## 3. 실습 — [무엇을 만든다]

[개념 직후 곧바로 실행 가능한 최소 코드. 전체 스크립트는 practice/ 에 두고, 본문에는 핵심 조각을 보여준다.]

```python
# practice/[파일명].py 의 핵심 부분
[실행 가능한 코드. import·전제·주석 포함]
```

> 전체 코드와 실행 절차는 [`practice/`](practice/) 와 [`labs/`](labs/) 참조.
> 비용을 줄이려면 [Ollama + bge-m3 대안 분기]를 따른다(stack-conventions 규약).

## 4. 결과 해석

[출력 예시와 의미. "이 숫자가 왜 중요한가."]

---

## 🚨 자주 하는 실수

1. **[실수 1]** — [왜 틀리는지 + 올바른 방법]
2. **[실수 2]** — [...]
<!-- 1~3개. 학습자가 실제로 빠지는 함정 위주. 일반론 금지. -->

## 출처

- [공식 docs / GitHub / 논문 — 제목, URL]
- [...]
<!-- roadmap "자료"·"참고 문헌"의 URL과 일치시킨다. 새 URL은 선별 검증 후 추가. -->

## 다음 토픽

→ [[다음 토픽 제목]](../[NN]-[topic-slug]/lesson.md)
<!-- 마지막 줄. 상대 경로. Phase 마지막 토픽이면 다음 Phase 첫 토픽으로 연결. -->
````

---

## 디렉토리 산출물 규약

```
course/phase-<NN>-<slug>/<NN>-<topic-slug>/
  ├─ lesson.md        # 위 템플릿 (= 사이트 한 페이지)
  ├─ practice/        # 실행 가능한 전체 코드: .py / .ipynb / Dockerfile / docker-compose.yml / .cypher / 매니페스트
  ├─ labs/            # 단계별 핸즈온 명령 + 예상 출력 (README.md 또는 steps.md)
  └─ img/             # (선택) 본문 다이어그램. 본문은 이미지 없이도 성립해야 함
```

- **슬러그·번호는 [study-roadmap.md] 그대로.** 임의 변경 금지.
- 링크는 **상대 경로**, 이미지는 `./img/`. 사이트 빌더 전환 시 깨지지 않게.
- `labs/`에는 명령마다 **예상 출력**을 함께 적는다(학습자가 실행 결과를 대조할 수 있게).
